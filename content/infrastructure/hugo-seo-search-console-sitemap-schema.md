## Overview

Most Hugo SEO advice is either too generic (“install Search Console”) or too WordPress-shaped (“use Yoast”). QubitLogic does neither. The site uses **Hugo config + one custom head partial** to cover the essentials:

- `robots.txt` generated from `hugo.toml`
- `sitemap.xml` generated by Hugo
- Open Graph and Twitter cards
- JSON-LD from `layouts/partials/seo.html`
- Google Search Console verification from `GOOGLE_SITE_VERIFICATION`
- Sensible `noindex, follow` on taxonomy pages

That is enough for a technical content site. No plugin dashboard, no PHP, no hidden defaults.

If you already followed [Deploy Hugo to VPS: GitHub Actions & rsync](/infrastructure/deploy-hugo-github-actions-vps/), this guide adds the **SEO layer** on top of the same QubitLogic stack.

## Step 1 — Set the baseline in `hugo.toml`

For Hugo SEO, `hugo.toml` is where the site stops being a local prototype and becomes a crawlable public site. QubitLogic's relevant baseline looks like this:

```toml
baseURL = "https://qubitlogic.dev/"
locale = "en-gb"
enableRobotsTXT = true

[params]
  env = "production"
  title = "QubitLogic"
  description = "A technical workbench for Quantum-AI Hybrid Development. Code-first tutorials on Qiskit, AI agents, Python infrastructure, and self-hosted stacks."
  images = ["/images/og/home.png"]

[params.analytics.google]
  SiteVerificationTag = ""

[outputs]
  home = ["HTML", "RSS", "JSON"]

[security.funcs]
getenv = ['^HUGO_', '^CI$', '^GOOGLE_SITE_VERIFICATION$']
```

Why each line matters:

- `baseURL` controls canonical URLs, Open Graph image URLs, and sitemap entries.
- `enableRobotsTXT = true` makes Hugo emit `/robots.txt`.
- `home = ["HTML", "RSS", "JSON"]` keeps `/index.xml` for RSS and `/index.json` for site search.
- `getenv` explicitly allows the `GOOGLE_SITE_VERIFICATION` secret through Hugo's security model.

Hugo generates `sitemap.xml` by default. You do not need an extra plugin for that part.

Build locally and confirm the outputs exist:

```bash
hugo --buildFuture --minify
test -f public/robots.txt && echo "robots ok"
test -f public/sitemap.xml && echo "sitemap ok"
test -f public/index.xml && echo "rss ok"
```

If any of those fail, fix `baseURL`, outputs, or your build command before touching Search Console.

### Common `baseURL` mistakes

| `baseURL` value | What breaks |
|-----------------|------------|
| `http://localhost:1313/` | Canonicals and sitemap point at localhost |
| `https://example.com` (no trailing slash) | Nested relative URLs can behave unpredictably |
| Wrong subdomain (`www` vs apex) | Duplicate indexing and redirect churn |

Rule: **match the final public URL exactly**, including `https://` and trailing `/`.

## Step 2 — Use one SEO partial for robots, OG, and schema

QubitLogic keeps the SEO logic in `layouts/partials/seo.html`. That partial handles three jobs:

1. **Robots directives**
2. **Open Graph / Twitter card tags**
3. **JSON-LD schema for each page type**

The key pattern is the page-type split:

```go-html-template
{{ $isTaxonomy := or (eq .Kind "taxonomy") (eq .Kind "term") }}
{{ $isArticle := and (not .IsHome) .IsPage (ne .File.Path "about.md") }}

{{ if $isTaxonomy }}
<meta name="robots" content="noindex, follow">
{{ else }}
<meta name="robots" content="index, follow, max-snippet:-1, max-image-preview:large, max-video-preview:-1">
{{ end }}
```

That is a sensible default for a small site:

- **Articles** stay indexable
- **Section hubs** stay indexable
- **Taxonomy archives** stay crawlable but not indexable

This prevents tag and series archives from soaking up crawl budget or competing with the canonical article pages.

### QubitLogic schema coverage

| Page type | Schema emitted by `seo.html` |
|-----------|------------------------------|
| Home | `WebSite` + `Organization` |
| Section hub | `CollectionPage` + `BreadcrumbList` |
| About page | `AboutPage` + `BreadcrumbList` |
| Article | `TechArticle` + `BreadcrumbList` |
| Article with `faq:` front matter | `FAQPage` |

The article schema block is explicit rather than magical:

```go-html-template
{{ if $isArticle }}
  {{ $article := dict
    "@context" "https://schema.org"
    "@type" "TechArticle"
    "headline" $pageTitle
    "description" ($desc | plainify | truncate 200)
    "datePublished" (.Date.Format "2006-01-02T15:04:05Z07:00")
    "dateModified" (.Lastmod.Format "2006-01-02T15:04:05Z07:00")
  }}
<script type="application/ld+json">{{ $article | jsonify | safeJS }}</script>
{{ end }}
```

Two practical notes:

- QubitLogic uses `TechArticle`, which fits infrastructure/tutorial content better than a generic `BlogPosting`.
- The partial takes the first value from `.Params.images`, so each article should set a proper Open Graph image in front matter.

### FAQ schema from front matter

Because the site already reads `.Params.faq`, each article can ship both:

- a visible **Frequently Asked Questions** section in the body
- machine-readable `FAQPage` schema in the head

That is why the front matter format matters. On QubitLogic, the same `faq:` list powers both editorial structure and schema output.

## Step 3 — Verify Google Search Console with `GOOGLE_SITE_VERIFICATION`

This is the cleanest part of the stack once you understand the flow.

1. In Search Console, add a property for `https://yourdomain.dev`
2. Choose **HTML tag**
3. Copy the `content="..."` value only
4. Store it as the GitHub Actions secret `GOOGLE_SITE_VERIFICATION`

QubitLogic's head partial reads either the config value or the environment variable:

```go-html-template
{{- $googleVerify := or site.Params.analytics.google.SiteVerificationTag (getenv "GOOGLE_SITE_VERIFICATION") -}}
{{- with $googleVerify }}
<meta name="google-site-verification" content="{{ . }}">
{{- end }}
```

That gives you two valid options:

- **Preferred for CI**: keep the token in a secret
- **Fallback**: hard-code `SiteVerificationTag` in `hugo.toml`

If you want to test locally before pushing, build like this:

```bash
export GOOGLE_SITE_VERIFICATION="paste-search-console-token-here"
hugo --buildFuture --minify
rg 'google-site-verification' public -n
```

Expected result: the generated HTML contains a `<meta name="google-site-verification" ...>` tag on the homepage.

### Why this is better than hard-coding for most repos

| Approach | Good | Bad |
|----------|------|-----|
| Secret via `GOOGLE_SITE_VERIFICATION` | Token stays out of content commits | Needs CI secret management |
| `SiteVerificationTag` in `hugo.toml` | Simple, visible, no CI dependency | Token lives in the repo |

For QubitLogic, the secret route makes sense because the repo already builds via GitHub Actions.

## Step 4 — Submit `sitemap.xml` and inspect important URLs

Once the verification tag is live and Search Console confirms the property, submit the sitemap:

```text
https://yourdomain.dev/sitemap.xml
```

Then check the public outputs yourself:

```bash
curl -s https://yourdomain.dev/robots.txt
curl -s https://yourdomain.dev/sitemap.xml | rg "<loc>"
curl -s https://yourdomain.dev/index.xml | rg "<item>|<title>"
```

What to look for:

- `robots.txt` returns `200`
- `sitemap.xml` lists the important canonical URLs
- `index.xml` contains the latest posts if you expose RSS publicly

### URL Inspection workflow

Use **URL Inspection** for three types of pages:

1. A key article you want indexed immediately
2. A section page such as `/infrastructure/`
3. A taxonomy page such as `/series/phase-1-infrastructure/`

Expected outcomes on the QubitLogic pattern:

| URL type | Expected result |
|----------|-----------------|
| Article | Indexable |
| Section hub | Indexable |
| Taxonomy / term | `noindex, follow` |

If Search Console already discovered old taxonomy URLs before you added `noindex`, inspect them and request recrawl. QubitLogic's deployment notes even call out optional removal for archive-style URLs when needed.

## Step 5 — Internal linking is the multiplier

You can have perfect schema and still underperform if new posts are effectively orphaned.

QubitLogic uses a straightforward internal-linking pattern:

- Link each guide to its **prerequisite**
- Link to the **next step** in the series
- Use descriptive anchor text, not “click here”
- Link from newer posts back to cornerstone tutorials

You can see that pattern across the infrastructure series:

- [Ubuntu 24.04 VPS Hardening](/infrastructure/secure-ubuntu-24-04-vps-hardening/)
- [How to Provision a VPS for AI Agent Workloads](/infrastructure/how-to-provision-vps-ai-agent-workloads/)
- [Deploy FastAPI with Nginx and systemd](/infrastructure/deploy-fastapi-ubuntu-24-04-nginx-systemd/)
- [Deploy Hugo to VPS: GitHub Actions & rsync](/infrastructure/deploy-hugo-github-actions-vps/)

That helps readers, but it also creates a clean crawl path for new content.

### Simple internal-link audit

Search your content for direct infrastructure links:

```bash
rg '\]\(/infrastructure/' content -n
```

Then ask:

- Does each new article link to at least one older cornerstone post?
- Does it point to a sensible next step?
- Are the anchor texts descriptive?

For a small technical site, that matters more than endlessly tweaking keyword densities.

## Verification checklist

Run these checks after every SEO-related change:

| Check | Command | Expected |
|-------|---------|----------|
| Build succeeds | `hugo --buildFuture --minify` | No errors |
| Robots exists | `curl -s https://yourdomain.dev/robots.txt` | Valid response |
| Sitemap exists | `curl -s https://yourdomain.dev/sitemap.xml \| rg "<loc>"` | Canonical URLs present |
| Verification tag present | `curl -s https://yourdomain.dev/ \| rg "google-site-verification"` | Meta tag present |
| Schema present | `curl -s https://yourdomain.dev/infrastructure/your-post/ \| rg "application/ld\\+json"` | JSON-LD block present |

If the build is green but the live site is wrong, the usual culprit is deploy caching, not Hugo itself.

## Troubleshooting

**`sitemap.xml` is missing** — Confirm you are building the root Hugo site, not only a section or a subdirectory.

**Search Console cannot verify the site** — Make sure the meta tag is on the **homepage HTML**, not only in local development. Rebuild after setting the secret.

**Taxonomy pages still appear in Search results** — They may have been indexed before `noindex` shipped. Inspect the URL again and wait for recrawl.

**Open Graph image is wrong** — Add `images: ["/images/og/your-article.png"]` in the article front matter. QubitLogic's partial uses the first image from `.Params.images`.

**Rich results do not show immediately** — Schema presence is necessary, not sufficient. Google decides when to use it.

## Frequently Asked Questions

### Does Hugo generate robots.txt and sitemap.xml without a plugin?

Yes. QubitLogic enables `enableRobotsTXT = true` in `hugo.toml`, which makes Hugo emit `/robots.txt`, and Hugo generates `/sitemap.xml` automatically as long as `baseURL` is correct. You do not need Yoast, Rank Math, or a WordPress-style SEO plugin for those basics.

### Should Hugo taxonomy pages be indexed?

Usually no for a small technical site. QubitLogic's `layouts/partials/seo.html` marks taxonomy and term pages as `noindex, follow`, while keeping articles and section hubs indexable. That preserves crawl paths without cluttering Search Console with thin archive pages.

### Is `GOOGLE_SITE_VERIFICATION` better than hard-coding the Search Console token?

For most private repos, yes. QubitLogic allows `GOOGLE_SITE_VERIFICATION` through Hugo's `security.funcs.getenv` allow-list, then reads it in `layouts/_partials/head.html`. That keeps the verification string out of article commits while still baking the meta tag into the final HTML.

### Do I need a separate schema plugin for Hugo?

No if you are willing to own a partial. QubitLogic's `seo.html` already emits `BreadcrumbList`, `WebSite`, `Organization`, `CollectionPage`, `AboutPage`, `TechArticle`, and `FAQPage` JSON-LD based on page type and front matter.

### What internal linking matters most for Hugo SEO?

Link related guides in series order with descriptive anchor text. On QubitLogic, infrastructure posts point readers from VPS hardening to provisioning, then to FastAPI, Hugo deploys, Cloudflare, and the newsletter stack. Those links help both users and crawlers discover the important pages quickly.
