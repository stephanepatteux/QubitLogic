## Overview

Phase 2 of this series has benchmarked quantum algorithms honestly across six articles. The pattern is consistent: classical algorithms are faster and often more accurate on the problem sizes tractable on today's hardware.

This final Phase 2 article answers the question that pattern raises: **is quantum machine learning worth pursuing at all, and if so, under what specific conditions?**

The answer is not "yes" or "no" — it is a decision framework. QML has a legitimate theoretical advantage under conditions that are rarely met in practice today but will become increasingly relevant as hardware scales. Knowing those conditions prevents both premature adoption and premature dismissal.

---

## The Theoretical Case for QML

### Quantum kernel advantage

The strongest theoretical result for QML is the **quantum kernel advantage**: for certain datasets, a quantum feature map encodes data into a Hilbert space that is exponentially larger than any efficiently computable classical kernel space. A quantum SVM operating in this space can, in principle, separate classes that no classical kernel SVM can.

The key word is "certain datasets." The advantage requires:

1. **The data has a quantum-native structure** — i.e., it was generated by a quantum process or has correlations that map naturally to quantum states
2. **The classical simulation of the quantum kernel is hard** — i.e., no efficient classical approximation exists

Liu et al. (2021) proved a formal separation between quantum and classical kernels exists — but the constructed dataset was artificial. No natural dataset with this property has been identified in production.

### The dequantization threat

For every quantum speedup claim in ML, ask: can this be **dequantized**?

Dequantization (Tang et al., 2018 onwards) showed that several quantum ML algorithms — quantum PCA, quantum recommendation systems, quantum linear systems — could be matched by classical algorithms using **sampling access** to the data. The classical algorithms are not as fast in the absolute sense, but they are polynomial rather than exponential, eliminating any asymptotic quantum advantage.

The quantum kernel SVM from [Post 11](/quantum-coding/qiskit-scikit-learn-hybrid-workflow/) has not been dequantized for all cases — but the gap is narrowing.

## The Decision Framework

Before implementing any QML component, run through this checklist:

### Condition 1: Is your dataset high-dimensional with quantum-native structure?

```
Low-dimensional classical data (≤100 features)?  → Classical ML wins. No QML.
Text, images, tabular data?                        → Classical ML wins. No QML.
Quantum chemistry data (molecular energy levels)?  → QML is viable.
Quantum hardware calibration data?                 → QML is viable.
Simulated quantum circuit outputs?                 → QML is viable.
High-energy physics collision data?                → QML is potentially viable.
```

{{< callout type="tip" title="The clearest current use case" >}}
Quantum chemistry simulation is the strongest near-term application for QML. Training a variational quantum eigensolver (VQE) to predict molecular ground state energies exploits the quantum Hilbert space in a way that is genuinely hard to dequantize. If your data comes from molecular dynamics simulations, quantum sensors, or quantum hardware characterisation, QML is worth serious evaluation.
{{< /callout >}}

### Condition 2: What is your sample count?

Quantum kernel methods require evaluating the kernel matrix for every pair of training samples. This is `O(n²)` kernel evaluations, each requiring circuit execution.

| Training samples | Quantum kernel cost | Recommendation |
|---|---|---|
| < 100 | ~10,000 circuits | Feasible on simulator |
| 100–1,000 | 10K–1M circuits | Slow on simulator, expensive on hardware |
| > 1,000 | > 1M circuits | Not practical on current hardware |

Classical kernel SVMs scale to millions of samples. Until quantum hardware delivers shots at microsecond speeds, QML is bounded to small datasets.

### Condition 3: Can you measure a quantum advantage on your specific data?

Do not assume — test. The code from [Post 11](/quantum-coding/qiskit-scikit-learn-hybrid-workflow/) gives you the sklearn-compatible wrapper. Run this test:

```python
from sklearn.model_selection import cross_val_score
from sklearn.svm import SVC
from hybrid.quantum_kernel_svm import QuantumKernelSVM

X, y = load_your_data()   # your actual dataset

classical_scores = cross_val_score(
    SVC(kernel="rbf"), X, y, cv=5, scoring="accuracy"
)
quantum_scores = cross_val_score(
    QuantumKernelSVM(n_qubits=X.shape[1], feature_map_reps=2),
    X, y, cv=5, scoring="accuracy"
)

gap = quantum_scores.mean() - classical_scores.mean()
print(f"Classical SVM: {classical_scores.mean():.3f} ± {classical_scores.std():.3f}")
print(f"Quantum SVM:   {quantum_scores.mean():.3f} ± {quantum_scores.std():.3f}")
print(f"QML advantage: {gap:+.3f}")
```

If `gap <= 0`, classical SVM wins. Stop. Do not add quantum complexity for no gain.

## The Hardware Threshold

For QML to be practically useful — not just theoretically interesting — hardware needs to reach:

| Requirement | Current best (2026) | Required for QML advantage |
|---|---|---|
| Qubit count | 433 (IBM Osprey) | ~100–1000 (problem-dependent) |
| 2Q gate fidelity | 99.5% (IonQ Forte) | > 99.9% for deep circuits |
| Coherence time | ~100–500 µs | > 1 ms for iterative algorithms |
| Shot throughput | ~1,000 shots/s | > 1M shots/s for practical training |
| Error correction | Logical qubits: ~1,000 physical/logical | Surface code threshold |

We are within 2–5 years of crossing the shot throughput threshold on some platforms. The fidelity threshold is close for trapped-ion systems. The error correction threshold is a decade away for full fault tolerance.

**Practical implication:** invest time now in learning QML theory and tooling. Build the classical baselines. Set up the evaluation framework. When hardware crosses the threshold, you will be ready to deploy rather than starting from scratch.

## QML Tools Worth Knowing in 2026

### PennyLane — best for QML research

```bash
pip install pennylane pennylane-qiskit torch
```

PennyLane integrates quantum circuits directly into PyTorch autograd — you can backpropagate through a quantum circuit the same way you backpropagate through a neural network layer.

```python
import pennylane as qml
import torch

dev = qml.device("default.qubit", wires=4)

@qml.qnode(dev, interface="torch")
def quantum_layer(inputs, weights):
    qml.AngleEmbedding(inputs, wires=range(4))
    qml.BasicEntanglerLayers(weights, wires=range(4))
    return [qml.expval(qml.PauliZ(i)) for i in range(4)]


class HybridModel(torch.nn.Module):
    def __init__(self):
        super().__init__()
        self.q_weights = torch.nn.Parameter(torch.randn(3, 4))
        self.classical  = torch.nn.Linear(4, 2)

    def forward(self, x):
        q_out = torch.stack([quantum_layer(xi, self.q_weights) for xi in x])
        return self.classical(q_out)
```
